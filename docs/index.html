Choose Method 1 to install the global CLI tool, or Method 2 to add it as a standard Java library dependency.

Quick Start

Your first plugin in
5 steps

01

Scaffold a new plugin project

After installing PlugCLI, you have two ways to generate a project:

# Option A: Create a new folder automatically
plugcli init my-awesome-plugin

# Option B: Scaffold directly in your current folder
plugcli init .

This creates the entire folder structure, a sample command, build script, and README instantly.

02

Write your command class

Open src/main/java/myplugin/SampleCommand.java and write your logic:

package myplugin;

import com.cli.Plugin;
import com.cli.Command;
import com.cli.ArgParser;
import com.cli.PlugLogger;

@Command(
    value       = "greet",
    version     = "1.0",
    description = "Prints a personalized greeting",
    author      = "Your Name",
    usage       = "greet <name>"
)
public class GreetCommand implements Plugin {

    @Override
    public String getCommand() { return "greet"; }

    @Override
    public void execute(String[] args) {
        ArgParser parser = new ArgParser(args);
        String name = parser.getString(0, "World");
        PlugLogger.info("Hello, " + name + "! 👋");
    }
}
03

Build the plugin JAR

Run the included build script — it compiles and packages everything automatically:

cd my-awesome-plugin
.\build-plugin.ps1

# Output:
# Compiling plugin code...
# Creating JAR file...
# Build successful!
04

Run your command

The JAR lands in plugins/ and is instantly available:

plugcli greet Akshita

# [2026-04-25 12:00:00] INFO  Hello, Akshita! 👋
05

Run it asynchronously

Any command can run in the background — just add --async:

plugcli greet Akshita --async

# [INFO] Command 'greet' is running in background.
API Reference

Core building blocks

interface Plugin

The contract every plugin must implement. Two methods:

public interface Plugin {
  String getCommand();       // return command name
  void execute(String[] args); // your logic here
}
annotation @Command

Metadata annotation that registers your command automatically:

@Command(
  value       = "hello",      // required: command name
  version     = "1.0",        // optional: defaults to "1.0"
  description = "Says hello", // optional: shown in help
  author      = "Akshita",    // optional: shown in conflict table
  usage       = "hello <name>" // optional: usage hint
)
class ArgParser

Flexible argument parsing utility:

ArgParser p = new ArgParser(args);

p.getString(0)           // positional arg as String
p.getString(0, "World")  // with default fallback
p.getInt(0)              // parse as integer
p.getDouble(0)           // parse as double
p.getBoolean(0)          // parse as boolean
p.hasFlag("--verbose")   // check flag presence
p.getNamed("--name")     // get --name=value
p.size()                 // total arg count
class PlugLogger

ANSI-colored, timestamped logging:

PlugLogger.info("Task complete");    // 🟢 GREEN
PlugLogger.warn("Disk low");         // 🟡 YELLOW
PlugLogger.error("Crash!", ex);      // 🔴 RED
PlugLogger.debug("Loop count: 5");   // 🔵 CYAN (debug mode)

// Enable debug mode:
// java -jar plugcli.jar myCmd --debug
class HookRegistry

Middleware system — run code before/after commands:

HookRegistry hooks = engine.getHookRegistry();

// Run before EVERY command
hooks.addGlobalBeforeHook(args -> {
  System.out.println("Started: " + LocalTime.now());
});

// Run after a SPECIFIC command
hooks.addAfterHook("backup", args -> {
  System.out.println("Backup logged to audit trail");
});
file plugcli.config

Define command aliases — no code needed:

# plugcli.config
# Format: alias = command-name

b     = backup
t     = test
greet = hello
m     = add

Now plugcli b runs the backup command.

Built-in Commands

Framework commands

These commands are baked into the PlugCLI framework core — always available.

Command Usage Description
help plugcli help Lists all registered commands with descriptions, versions, and authors.
init plugcli init <name> Scaffolds a new plugin project — folder structure, sample command, build script, pom.xml.
switch plugcli switch <cmd> When multiple plugins register the same command, lets you switch the active version.
--debug plugcli <cmd> --debug Enables verbose debug logging for any command execution.
--async plugcli <cmd> --async Runs the specified command in a background thread (non-blocking).

Maven Integration

Auto-deploy plugins
on every build

Add this to your plugin's pom.xml so the built JAR is automatically placed in the plugins/ folder that PlugCLI reads from.

<build>
  <plugins>
    <plugin>
      <groupId>org.apache.maven.plugins</groupId>
      <artifactId>maven-jar-plugin</artifactId>
      <version>3.3.0</version>
      <configuration>
        <outputDirectory>${project.basedir}/plugins</outputDirectory>
      </configuration>
    </plugin>
  </plugins>
</build>

Then just run mvn clean package and your plugin is live. ⚡

Reference

Built-in Commands & Flags

PlugCLI comes with powerful global commands and flags available out of the box.

plugcli help

Lists all registered commands currently available in your plugins/ folder, along with their descriptions and authors.

plugcli init <name> or plugcli init .

Scaffolds a complete, ready-to-build plugin project. Use . to scaffold directly in the current directory.

plugcli switch <command>

If multiple plugins register the exact same command, this lets you choose which version should be executed by default.

--async

Append this global flag to any command to execute it in a background thread instantly. (e.g., plugcli greet --async).

--debug

Append this global flag to enable verbose framework logging, perfect for tracking down plugin loading issues.
